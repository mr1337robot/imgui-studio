# ADR 0001: Use Real Dear ImGui Compiled to WebAssembly for Canonical Preview

**Status:** Accepted  
**Date:** July 9, 2026  
**Decision owners:** Studio architecture  
**Related:** `PRD.md`, `TECHNICAL_DESIGN.md`, `TEST_PLAN.md`

## Context

The product exists because agents need a rapid visual loop for custom Dear ImGui C++. An HTML/CSS recreation, layout DSL, or screenshot mockup would create a second rendering implementation and could not reliably represent custom `ImDrawList` geometry, ImGui item behavior, font atlas inputs, clipping, IDs, or animation state. Export fidelity would become an ongoing translation problem.

The preview must also produce structured inspection and deterministic screenshots while remaining isolated from the parent Studio application.

## Decision

The canonical browser preview will execute the project's real C++ menu/widget source with the pinned Dear ImGui and Studio runtime compiled by Emscripten to WebAssembly. The browser host renders through the selected WebGL2 backend and runs in a sandboxed iframe/dedicated preview origin.

Browser and Windows native targets compile the same project translation units. Platform/bootstrap/renderer hosts differ, but user menu/widget implementations and portable runtime APIs do not.

A successful preview identity includes project revision, immutable build ID, preview instance ID, runtime protocol version, asset/font configuration digest, viewport, and DPI. HTML representations may be used for Studio controls or accessibility around the canvas, but never as the canonical menu render.

## Consequences

### Positive

- What the agent sees is produced by real Dear ImGui and custom C++ drawing.
- Preview/export drift is bounded to platform/backend/font rasterization differences rather than a lossy translation layer.
- ImGui interaction, clipping, draw commands, and internal adapter behavior can be tested directly.
- The same code participates in browser/native parity and package verification.

### Negative

- Source changes require compilation and WASM relinking; build latency is a product constraint.
- Browser execution needs an explicit protocol, asset packaging, screenshot readback, and crash recovery.
- Browser/native pixel identity is not guaranteed across renderers, so structured geometry and perceptual comparisons are both required.
- Preview isolation must be maintained despite running user-authored logic.

## Alternatives considered

- **HTML/CSS recreation:** fastest web iteration, rejected because it is not authoritative for custom ImGui widgets or native export.
- **JSON layout interpreted by ImGui:** deterministic but constrains arbitrary C++ authoring and creates a generator/runtime abstraction the product explicitly avoids.
- **Remote native screenshot service:** can render real code but adds platform provisioning, latency, and unsafe hosted compilation before the local MVP proves value.
- **Native desktop-only preview:** accurate but denies agents the browser automation, capture, and isolation loop central to the product.

## Implementation constraints

- Cache stable Dear ImGui/backend/runtime objects and unchanged project units.
- Replace previews only after compile, link, artifact validation, and smoke initialization succeed.
- Retain the prior successful preview after failure.
- Capture the underlying RGBA8 framebuffer without browser chrome or CSS scaling.
- Prevent preview access to parent DOM/storage, arbitrary network, and local filesystem as specified in `SECURITY_MODEL.md`.
- Keep project code backend-agnostic for portable-tier exports.

## Acceptance criteria

- One custom widget source file containing no stock visible checkbox implementation compiles unchanged into WASM and Windows native fixtures.
- Browser output is generated by Dear ImGui draw data and is manually/scriptably interactive.
- Every capture is attributable to an immutable build/revision and asset configuration.
- Browser/native widget geometry differs by no more than two pixels in the fixed parity fixture.
- A failed replacement build leaves the previous preview running and visibly stale.
- Tests prove the preview cannot read an external test file or use undocumented parent/network channels.
